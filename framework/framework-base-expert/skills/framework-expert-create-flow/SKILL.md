# Framework Expert Create Flow

An interactive wizard for creating a complete, standards-compliant connsys-jarvis Expert. The only required file is `expert.json`; `soul.md` and `expert.md` are optional. Skills are the primary mechanism for defining Expert capabilities. This skill walks through folder scaffolding, validates naming, and ends with a checklist.

> **Key principle**: Skills should be the primary mechanism for Expert capabilities. `soul.md` and `expert.md` are optional identity/behavior files. Focus effort on creating well-structured Skills.

---

## Step 0: Prerequisite Check

Before writing any files, collect the following. Ask if not provided — don't guess.

1. **Domain** — which domain does this expert belong to? (e.g., `wifi-bora`, `sys-bora`, `framework`)
2. **Expert name** — what is the purpose? (full name will be `{domain}-{purpose}-expert`)
3. **Is it a base expert?** — shared resource container (`is_base: true`) or a specific-task expert?
4. **Owner team** — which team maintains it? (e.g., `wifi-bora-team`)
5. **Dependencies** — which existing experts does it depend on? (e.g., `framework-base-expert`, `wifi-bora-base-expert`)
6. **Core purpose** — one sentence: what does this expert enable Claude to do?

Propose the full expert name and confirm with the user before creating any files.

## connsys-jarvis Expert Naming Rules

**Format**: `{domain}-{purpose}-expert`

| Part | Rules | Examples |
|------|-------|---------|
| `domain` | Layer 1 domain prefix | `wifi-bora`, `sys-bora`, `bt-bora`, `framework` |
| `purpose` | kebab-case description | `memory-slim`, `preflight`, `cr-robot`, `base` |
| suffix | always `-expert` | — |

**Special case — Base Expert**:
- Every domain must have exactly one `{domain}-base-expert`
- Set `is_base: true` in expert.json
- No dependencies (other experts depend on it, not the other way)
- Acts as a shared skill/hook/command container for its domain

**Expert directory path**: `connsys-jarvis/{domain}/{domain}-{purpose}-expert/`

## Step 1: Create the Directory Structure

Scaffold the full Layer 4 structure:

```
{domain}/{domain}-{purpose}-expert/
├── expert.json     ← Manifest (REQUIRED — the only required file)
├── soul.md         ← Identity & personality (optional)
├── expert.md       ← Key behaviors & tools (optional, loaded into CLAUDE.md)
├── README.md       ← Traditional Chinese documentation (required)
├── skills/         ← Skill folders — PRIMARY mechanism (use framework-skill-create-flow)
├── hooks/          ← Lifecycle shell scripts (optional)
├── agents/         ← Sub-agent definitions (optional)
└── commands/       ← Slash command definitions (optional)
```

### Expert = Claude Code Plugin folder structure

The Expert directory doubles as a Claude Code Plugin package. The full structure:

```
{domain}/{expert-name}/
├── .claude-plugin/
│   └── plugin.json              # Generated by create_plugin_from_expert.py
├── expert.json                  # Required: Expert metadata
├── soul.md                      # Optional: Methodology and behavior
├── expert.md                    # Optional: Technical constraints
├── skills/                      # PRIMARY: Cross-tool compatible knowledge
│   └── {domain}-{name}-{type}/
│       └── SKILL.md             # Required frontmatter: name, description, version
├── commands/                    # Flat .md files (Claude Code slash commands)
│   └── {expert-name}-{action}-cmd.md
├── agents/                      # Flat .md files (subagent definitions)
│   └── {agent-name}.md
├── hooks/                       # Claude Code plugin hooks
│   ├── hooks.json               # Event binding configuration
│   └── scripts/
│       └── {expert-name}-{action}-hook.sh
├── rules/                       # Flat .md files → symlink to .claude/rules/
│   └── {expert-name}-{topic}-rule.md
├── scripts/                     # Helper scripts
└── README.md
```

> **Skills are the PRIMARY mechanism** because they work across AI tools (Claude Code, Cursor, Windsurf, etc.). Commands, hooks, rules, and agents are for Claude Code Plugin compatibility only.

### Naming conventions

| Asset | Pattern | Notes |
|-------|---------|-------|
| skills | `{domain}-{name}-{type}/SKILL.md` | Subfolder. type = `flow` / `knowhow` / `tool` |
| commands | `{expert-name}-{action}-cmd.md` | Flat file (e.g., `framework-base-expert-list-cmd.md`) |
| hooks | `hooks/scripts/{expert-name}-{action}-hook.sh` | Flat file under `scripts/` |
| rules | `{expert-name}-{topic}-rule.md` | Flat file |
| agents | `{agent-name}.md` | Flat file |

### Metadata headers

**Command metadata:**
```yaml
---
name: command-short-name
description: Command description
argument-hint: "[args]"
allowed-tools: Read, Bash(*)
---
```

**Skill metadata:**
```yaml
---
name: skill-short-name
description: When to use this skill
version: 1.0.0
---
```

**Agent metadata:**
```yaml
---
name: agent-short-name
description: Agent role and expertise
capabilities:
  - Specific task 1
---
```

> **重要**：`name` 欄位決定使用者能否用短名呼叫。Plugin 安裝後全名為 `pluginName:name`（如 `framework-base-expert:framework-skill-create-flow`），但 Claude Code 會用 frontmatter `name` 做匹配，所以使用者只需輸入 `/framework-skill-create-flow` 即可呼叫。**務必在所有 SKILL.md / command.md / agent.md 的 frontmatter 加上 `name` 欄位。**

Create the directory, then proceed file by file.

## Step 2: Write expert.json

### Schema

```json
{
  "name": "{domain}-{purpose}-expert",
  "display_name": "{Human Readable Name}",
  "domain": "{domain}",
  "owner": "{domain}-team",
  "description": "{one-sentence description}",
  "version": "1.0.0",

  // Base expert only — omit for regular experts:
  "is_base": true,

  // Triggers: keywords that suggest this expert should be active
  "triggers": ["keyword1", "keyword2"],

  // State-machine transitions: when condition X is met, suggest switching to expert Y
  "transitions": {
    "ANALYSIS_DONE": "{domain}-{next}-expert"
  },

  // Dependencies: which experts' assets to symlink into .claude/
  "dependencies": [
    {
      "expert": "framework/framework-base-expert",
      "skills": ["framework-expert-discovery-knowhow", "framework-memory-tool"],
      "hooks": ["all"],
      "agents": [],
      "commands": ["framework-experts-tool"]
    }
    // Add domain base-expert dependency if applicable
  ],

  // Internal: this expert's own skills/hooks/agents/commands
  "internal": {
    "skills": [],
    "hooks": [],
    "agents": [],
    "commands": []
  },

  // Scenarios this expert supports
  "scenarios": ["agent-first", "legacy"],

  // Operations requiring human confirmation before execution
  "human_in_the_loop": {
    "require_confirm": ["git push", "git reset --hard", "rm -rf"]
  },

  // Regex patterns for symlinks to exclude after creation
  "exclude_symlink": {
    "patterns": []
  }
}
```

### Dependencies selection rules

| Situation | Write | Effect |
|-----------|-------|--------|
| Inherit all | `"skills": ["all"]` | Symlink every skill from that expert |
| Explicit list | `"skills": ["foo-tool", "bar-flow"]` | Only symlink named skills |
| Omit key | (don't write `"skills"`) | Inherit nothing |

> `hooks`, `agents`, `commands` follow the same rule independently.

> **Plugin install mode limitation**: In plugin install mode, dependency skill selection is not available — all dependency plugin skills are loaded. This is a Claude Code plugin system limitation. Selective skill loading only works in setup.py symlink mode.

## Step 3: Write soul.md (Optional)

soul.md defines **who this expert is**. Write in the voice of the expert itself. This file is optional — skip if the Expert's identity is sufficiently defined by its skills.

**Recommended sections** (if creating soul.md):

```markdown
# {ExpertName} — Soul

## Identity

{One clear sentence: who is this expert and what is its singular purpose?}

## Communication Style

{How does it communicate? Examples:}
- Direct and technical — uses precise terminology, no fluff
- Collaborative — explains reasoning, invites confirmation before acting
- Language: Traditional Chinese (technical terms remain in English)

## Values & Principles

{3–5 core values. Each should reflect what the expert genuinely prioritizes.}
- **{Value}**: {Why this matters for this expert's domain}

## Domain Expertise

{What does this expert know deeply?}
- {Specific domain knowledge 1}
- {Specific domain knowledge 2}

## Personality

{1–2 sentences describing disposition and working style.
Example: "Proactive but aware of its own boundaries. Asks rather than guesses when uncertain."}

## Collaboration Style

{How does it work with the engineer?}
- When to ask vs. act autonomously
- Tone and escalation behavior
```

**Tips**:
- Identity = one sentence, no ambiguity
- Values should differ from other experts — what is *unique* about this expert's priorities?
- Domain expertise is factual, not generic ("Wi-Fi 802.11 protocol stack" not "software development")

## Step 4: Write expert.md (Optional)

expert.md is the **public interface** of this expert — setup.py @includes this into CLAUDE.md. Claude reads this every session. Keep it factual and current.

**Required sections**:

```markdown
# {Expert Display Name}

## Overview

{2–3 sentences: what this expert does, when to use it, what domain it covers}

## Key Behaviors

- {Observable behavior 1}
- {Observable behavior 2}
- Example: "Automatically reads the latest memory summary at session start"

## Tools Available

| Command | Description |
|---------|-------------|
| `/{command}` | {what it does} |

## Skills

| Skill | Type | Description |
|-------|------|-------------|
| `{domain}-{name}-{type}` | {type} | {one-line description} |

## Hooks

| Hook | Trigger | Description |
|------|---------|-------------|
| `{hook}.sh` | {when} | {what it does} |

## Environment Variables

| Variable | Description |
|----------|-------------|
| `CONNSYS_JARVIS_*` | {description} |

## Memory Structure (if applicable)

{Describe what this expert reads/writes to .connsys-jarvis/memory/}
```

**Tips**:
- This file is loaded every session — keep it concise and accurate
- Skills and Hooks tables should stay in sync with expert.json internal list
- Do not duplicate soul.md content here — expert.md is behavioral, not identity

## Step 4.5: Create using-knowhow Skill (Conditional — Optional)

> **Skip this step if**: The expert has neither soul.md nor expert.md (skills-only expert).
> soul.md and expert.md are **optional**, therefore this using-knowhow skill is also **optional**.

If you created soul.md and/or expert.md in the previous steps, create a companion `{expert-name}-using-knowhow` skill that merges their content. This ensures the expert's guiding principles are available in **all installation modes** (setup.py, Plugin Marketplace, npx skills).

**Why this is needed**: Plugin Marketplace and npx skills do **NOT** install CLAUDE.md. Without the using-knowhow skill, soul.md and expert.md content would be lost in those modes. See `doc/agents-design.md` §2.8 for full rationale.

### Create the skill directory

```
skills/{expert-name}-using-knowhow/
├── SKILL.md
└── README.md
```

### SKILL.md template

```yaml
---
name: {expert-name}-using-knowhow
description: "MANDATORY foundation for all {domain} skills. You MUST read this skill FIRST before using ANY skill prefixed with '{domain}'. Contains the expert's methodology, values, critical constraints, and engineering rules. Once read in a session, no need to re-read."
version: "1.0.0"
---

# {Expert Display Name} — Using Knowhow

> This skill contains the guiding principles for all {domain} operations.
> Extracted from soul.md + expert.md to ensure availability across all installation modes.

---

## Identity & Methodology

{full content of soul.md}

---

## Technical Constraints & Capabilities

{full content of expert.md}
```

### After creating the using-knowhow skill

Simplify both soul.md and expert.md to a single HTML comment:

```html
<!-- Content migrated to {expert-name}-using-knowhow skill. Edit that skill's SKILL.md instead. See doc/agents-design.md §2.8. -->
```

### Register in expert.json

Add `"{expert-name}-using-knowhow"` to `internal.skills` (unless using `["ALL"]`).

### ⚠️ Risk

This design relies on model's skill description matching — not guaranteed 100% trigger rate. Use MANDATORY/MUST language in the description to maximize trigger probability.

## Step 5: Set Up Content Folders (Primary)

### skills/
Each skill follows `{domain}-{name}-{type}/` naming. Use `framework-skill-create-flow` to create each skill interactively — it handles SKILL.md, README.md, expert.json registration, naming validation, and eval setup.

```
skills/
└── {domain}-{name}-{type}/
    ├── SKILL.md
    └── README.md
```

Typical skill types for a new expert:
- `flow`: the primary SOP or workflow this expert executes
- `knowhow`: domain knowledge reference (architecture, protocol, code rules)
- `tool`: external tool operation guides

### hooks/
Shell scripts that run at lifecycle events. Shell-first; complex logic in `{name}-helper.py`.

```
hooks/
├── hooks.json                ← Event binding configuration (plugin mode)
├── scripts/
│   ├── {expert-name}-session-start-hook.sh
│   ├── {expert-name}-session-end-hook.sh
│   ├── {expert-name}-pre-compact-hook.sh
│   └── {expert-name}-mid-session-checkpoint-hook.sh
```

Only add hooks this expert genuinely needs. Base experts typically have all four.

> **Plugin mode**: Use `${CLAUDE_PLUGIN_ROOT}` for portable path references in hook scripts. This variable resolves to the plugin's install directory at runtime.

### agents/
Sub-agent prompt files. Each `.md` file defines a specialized subagent Claude can spawn.

```
agents/
└── {agent-name}.md    ← System prompt / instructions for the subagent
```

Use sub-agents for: parallel tasks, specialized analysis (log parsing, doc lookup), or operations that benefit from fresh context.

### commands/
Slash command definitions as flat `.md` files.

```
commands/
└── {expert-name}-{action}-cmd.md
```

Phase 1: inherit commands from framework-base-expert; only add new commands when the expert needs unique slash commands not available in its dependencies.

## Step 6: Write README.md

Write in **Traditional Chinese (Taiwan)**. Follow the template from `framework-skill-create-flow`'s README.md (sections: Introduction, Owner, Features, Goals, Design Philosophy, Risks, Alternatives, Source).

## Step 7: Register and Install

After all files are written:

1. **Check the directory path** is correct: `connsys-jarvis/{domain}/{expert-name}/`

2. **Install with setup.py** to verify the expert works end-to-end:
   ```bash
   # First-time install (replaces current expert):
   uv run ./connsys-jarvis/scripts/setup.py --init {domain}/{expert-name}/expert.json && source .connsys-jarvis/.env

   # Add alongside existing expert:
   uv run ./connsys-jarvis/scripts/setup.py --add {domain}/{expert-name}/expert.json && source .connsys-jarvis/.env
   ```

3. **Run doctor** to validate:
   ```bash
   uv run ./connsys-jarvis/scripts/setup.py --doctor
   ```

4. **Generate plugin** (if applicable):
   ```bash
   uv run ./connsys-jarvis/framework/framework-base-expert/skills/framework-expert-create-flow/scripts/create_plugin_from_expert.py
   ```
   > Run `create_plugin_from_expert.py` after creating an expert to generate `.claude-plugin/plugin.json` and `marketplace.json`.
   > The script auto-detects `skills/`, `commands/`, `agents/`, `hooks/hooks.json`, `.mcp.json`, `.lsp.json`, `monitors/monitors.json` in each expert directory.

## Step 8: Create Expert Checklist

Run through every item before considering the expert complete.

### A. Directory & Files

- [ ] Expert directory at correct path: `connsys-jarvis/{domain}/{domain}-{purpose}-expert/`
- [ ] `expert.json` — valid JSON, all required fields present
- [ ] `soul.md` — (optional) has Identity, Communication Style, Values, Domain Expertise, Personality, Collaboration Style
- [ ] `expert.md` — (optional) has Overview, Key Behaviors, Skills table, Hooks table
- [ ] `README.md` — in Traditional Chinese, has all required sections

### B. expert.json Validation

- [ ] `name` matches directory name exactly
- [ ] `domain` matches the domain prefix
- [ ] `version` is `"1.0.0"` for new experts
- [ ] `dependencies` lists `framework/framework-base-expert` (unless this IS framework-base-expert)
- [ ] `internal.skills` lists all skills in the `skills/` folder
- [ ] `internal.hooks` lists all hooks in the `hooks/` folder
- [ ] `is_base: true` if this is a base expert; field absent for regular experts
- [ ] `human_in_the_loop.require_confirm` covers all destructive operations in this domain

### C. Content Quality

- [ ] soul.md Identity is one clear sentence (not vague) — if soul.md exists
- [ ] expert.md Skills table matches `expert.json internal.skills` — if expert.md exists
- [ ] expert.md Hooks table matches `expert.json internal.hooks` — if expert.md exists
- [ ] Skills are the primary mechanism for Expert capabilities

### D. Skills

- [ ] Each skill created using `framework-skill-create-flow`
- [ ] Each skill name follows `{domain}-{name}-{type}` format
- [ ] Each skill registered in `expert.json internal.skills`

### E. Installation Validation

- [ ] `setup.py --doctor` passes all checks

### F. Requirements Alignment

- [ ] Expert follows naming rule from requirements §1.3 / design §2.4
- [ ] Base expert has `is_base: true`
- [ ] `human_in_the_loop` covers all operations flagged as risky in requirements
- [ ] `transitions` defined if this expert has a natural successor in the workflow

### G. using-knowhow Skill (if soul.md or expert.md exists — optional)

- [ ] `{expert-name}-using-knowhow` skill created with merged soul.md + expert.md content
- [ ] SKILL.md description references the domain prefix for automatic triggering
- [ ] soul.md simplified to HTML comment pointer (see Step 4.5)
- [ ] expert.md simplified to HTML comment pointer (see Step 4.5)
- [ ] Skill registered in `expert.json internal.skills` (unless using `["ALL"]`)
