# connsys-jarvis 功能化簡計畫

## Context

connsys-jarvis 目前的 Expert 結構過於複雜（6 個必要檔案），CLAUDE.md 生成邏輯依賴 "base expert" 概念，且與 Claude Code Plugin 系統不相容。本次重構目標：

1. **化簡 Expert 檔案結構**：從 4 個必要 MD → 2 個可選 MD（soul.md + expert.md）
2. **對齊 Claude Code Plugin 規範**：Expert = Plugin，可直接用 Marketplace 安裝
3. **移除 base-expert 概念**：多 Expert 時統一按安裝順序讀取
4. **Skills 優先策略**：跨工具相容（未來支援 opencode、gemini 等）
5. **提供 3 種安裝方式**：setup.py（預設）、Marketplace、npx skills

```python
def detect_scenario(workspace: Path) -> str:
    if (workspace / ".repo").exists() or (workspace / ".git").exists():
        return "legacy"
    # 檢查是否有其他資料夾（排除 .connsys-jarvis、connsys-jarvis 本身）
    for item in workspace.iterdir():
        if item.is_dir() and item.name not in (".connsys-jarvis", "connsys-jarvis"):
            return "legacy"
    return "agent-first"
```

## Phase 2: Expert MD 檔案化簡

### 2.1 新的 soul.md 結構（可選）

**寫作原則**：
- **不寫**「你是什麼專家」— 身份由 expert.json 的 name/description 決定
- **寫**方法論、做事方式、行為準則
- 能用 skill 就用 skill（跨 agent 工具通用）
- 參考風格：[Karpathy CLAUDE.md](https://github.com/multica-ai/andrej-karpathy-skills/blob/main/CLAUDE.md)

```markdown
# {Expert Name} — Soul

## Methodology
{做事方式：先確認再動手、最小改動原則、提問而非假設...}

## Values & Principles  
{3-5 條行為原則，含衝突解決優先級}

## Boundaries
{不做什麼 — 原 duties.md 的職責分離內容}

## Communication Style
{語言、語調、升級行為}
```

### 2.2 新的 expert.md 結構（可選）

**寫作原則**：
- **只寫常被遺忘的重要事情**（Claude Code 會特別關注 CLAUDE.md 內容）
- **不寫**「你是什麼專家」→ 寫有什麼能力、可以解決什麼問題
- **不重複 Skill description**（Claude Code 自動讀取 SKILL.md frontmatter 的 description，不需重複）
- 能用 skill 就用 skill（跨 agent 工具通用）
- 建議 50-200 行（過長會導致注意力稀釋）

**參考知識庫**：
- [CLAUDE.md Best Practices — Skills vs CLAUDE.md](https://github.com/swchen44/personal-knowledge-base-from-ai/blob/main/AI/2026-04-15-CLAUDE-MD-BEST-PRACTICES-EXPERT-GUIDE-SKILLS-VS-CLAUDEMD.md)
- [Skills vs Commands vs Subagents](https://github.com/swchen44/personal-knowledge-base-from-ai/blob/main/AI/2026-04-16-CLAUDE-CODE-SKILLS-VS-COMMANDS-VS-SUBAGENTS-COMPLETE-COMPARISON.md)

**四大原則**：

1. **寫規則，不寫介紹**
   - ✅ 在這個專案裡應該怎麼工作、哪些事不能亂做、哪些規則是程式碼推導不出來的
   - ❌ 專案多先進、目錄結構長什麼樣、一般性開發建議
   - expert.md 更像「工作約束」，不是「專案簡介」

2. **寫約束，不寫常識**
   - ❌ 「寫程式要注意可讀性」「記得測試」「不要提交密鑰」— 模型本來就知道，收益極低
   - ✅ 這個專案獨特的約束、這個團隊獨特的習慣、這個 repo 獨特的邊界

3. **寫長期有效的流程資訊，不寫臨時任務**
   - ❌ 這次需求怎麼做、當前任務分幾步、這輪開發計畫、這次改動細節
   - ✅ 長期有效的規則和流程（「長期有效」是關鍵判斷標準）

4. **出現兩次以上的錯誤模式，更新到 expert.md**
   - 模型重複犯的錯 → 寫進 expert.md 作為永久約束

**撰寫技巧**：
- 具體可執行的指令（非模糊的「要有幫助」）
- 正面陳述搭配範例（正確 vs 錯誤）
- 階層式規則（優先順序）
- 決策理由（為什麼這樣做）
- 常見工作流程（步驟序列）
- 回應風格（格式期望）
- 禁止項一定要搭配替代方案（「不要用 X → 改用 Y」）

### 2.3 內容遷移對照表

| 來源 | 目的地 | 原因 |
|------|--------|------|
| rules.md "Must Always" | Skills（knowhow 類型）| 跨工具相容 |
| rules.md "Must Never" | Skills（SKILL.md 中的約束條件）| 跨工具相容 |
| rules.md "Conflict Resolution" | soul.md "Values & Principles" | 身份層級 |
| duties.md "Primary Duties" | Skills（每個職責 → flow/knowhow skill）| Skills 優先 |
| duties.md "Segregation of Duties" | soul.md "Boundaries" | 身份層級 |
| duties.md "KPIs" | 刪除或移至 README.md | 非 AI 可執行 |

### 2.4 每個 Expert 的操作

對 9 個 Expert 各自：
1. 重寫 soul.md（合併原 conflict resolution + segregation）
2. 重寫 expert.md（遵循新格式 + 8 大技巧）
3. **直接刪除** rules.md 和 duties.md（有用內容先遷移到 soul.md / expert.md / skills）
4. 必要時將規則內容轉為新 skill

**修改檔案**：所有 9 個 expert 目錄下的 soul.md、expert.md

## Phase 3: create_plugin_from_expert.py

### 3.1 新檔案

`/scripts/create_plugin_from_expert/create_plugin_from_expert.py`

**功能**：
1. 掃描所有 `*/*/expert.json`
2. 每個 expert → 生成 `.claude-plugin/plugin.json`
3. 在 repo 根目錄生成 `marketplace.json`

### 3.2 expert.json → plugin.json 欄位映射

```
expert.json                    → plugin.json
─────────────────────────────────────────────
name                           → name
version                        → version  
description                    → description
owner                          → author.name
internal.skills[]              → skills (directory paths)
internal.hooks[]               → hooks
internal.commands[]            → commands (directory paths)
internal.agents[]              → agents (directory paths)
dependencies[].skills[]        → skills (額外 dependency skill paths)
```

### 3.3 依賴 Skill 路徑解析（關鍵複雜度）

plugin.json 的 `skills` 欄位需要列出所有相依 skill 的相對路徑：

```json
{
  "name": "wifi-bora-memory-slim-expert",
  "skills": [
    "./skills/",
    "../../framework/framework-base-expert/skills/",
    "../../wifi-bora/wifi-bora-base-expert/skills/"
  ]
}
```

- 遞迴走訪 dependencies[]
- `"skills": "all"` → 掃描該 dependency 的 skills/ 目錄
- `"skills": ["specific-skill"]` → 建構個別路徑
- 所有路徑必須以 `./` 開頭（plugin.json 規範）
- 驗證路徑存在性

### 3.4 marketplace.json 格式

```json
{
  "name": "connsys-jarvis",
  "owner": { "name": "connsys-team" },
  "plugins": [
    {
      "name": "framework-base-expert",
      "source": "./framework/framework-base-expert",
      "version": "1.0.0",
      "description": "Framework coordination expert"
    }
  ]
}
```

### 3.5 CLI 介面

```
python create_plugin_from_expert.py [OPTIONS]

  --doctor     用 claude --plugin-dir -p 驗證 plugin.json 正確性
               檢查 expert.json 與 plugin.json 是否一致
  --dry-run    產生設定檔但不寫入
  --verbose    開啟詳細 log
  --help       詳細使用說明
  --expert-dir 指定掃描根目錄（預設：connsys-jarvis repo root）
```

`--doctor` 模式（**claude CLI 為必要前置條件**）：
1. 靜態檢查：expert.json 中列出的 skill/hook/command 路徑是否存在
2. plugin.json 與 expert.json 的 skill 清單是否一致
3. 用 `claude --plugin-dir <path> -p "list skills"` 動態驗證每個 plugin 的 skill 載入
4. 找不到 claude CLI 時直接報錯退出（不降級為純靜態檢查）
